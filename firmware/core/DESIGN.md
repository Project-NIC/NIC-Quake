# NIC Core — Platform Design Decisions

The durable decision log for the **universal, front-agnostic NIC platform** — the
NodeBus, the clock backbone, the link/control protocols, addressing, timing and the
master uplink. These hold for *every* front (seismo, weather, iono, …); a front never
re-litigates them.

Front-specific decisions live next to their front — the seismograph ones in
`../DESIGN.md` (D2–D5, D8, D11, D14–D18, D21, D22). Original D-numbers are kept
so the cross-references between decisions still resolve.

---

## D1 — Modular hardware-agnostic libraries + thin glue

Firmware is a set of small portable-C `nq-*` libraries that reach hardware only
through callbacks, glued together by `nic-node` (the only STM32-specific part).
Every library compiles and unit-tests on a host PC. See `firmware/README.md`.

## D6 — Clock architecture: one master crystal, integer division, MCU-generated sensor clocks

- The master carries a **single 8.192 MHz crystal** — the only oscillator in the
  network. 8.192 = 8 × 1.024 MHz (the RTC/audio-clean value); every clock the
  node needs is an integer division of it. Swapping this one crystal retunes the
  whole network.
- The clock is distributed over RS-485 pair 2; the **THVD1450 regenerates the
  edges** at the far end, so it arrives as clean digital transitions.
- **Master-side gating: hard-wired oscillator, soft enable.** The 8.192 MHz
  oscillator output goes **straight to the clock-pair transceiver's driver input** —
  the shortest, lowest-jitter path; the clock never passes *through* the MCU (routing
  it through a GPIO would only add jitter, and the network's whole timing rides on this
  edge). The MCU drives only the transceiver **DE** (driver-enable): "start the clock"
  is asserting DE after the TCXO settles, and dropping DE kills the clock on the bus — a
  clean network-restart hammer (all nodes lose HSE → fall back to RC). The clock pair is
  dedicated and unidirectional, so nothing arbitrates it. *Optional seam:* a parallel tap
  from the oscillator to a spare MCU input lets the master discipline its **own**
  sub-second time to the network clock (the oscillator fans out to several inputs) — not
  required, since the per-second TICK + GPS PPS already anchor absolute time (D5; Iono).
- **Node HSE:** the distributed 8.192 MHz feeds the STM32H503 HSE directly
  (within the 4–50 MHz HSE range). If long-cable duty cycle ever drifts outside
  the PLL input window, drop in a single **divide-by-2 flip-flop** on the
  incoming clock — it outputs exactly 50/50 regardless of input duty (it toggles
  only on the rising edge; rising-edge spacing is the stable crystal frequency,
  the cable only distorts the 1/0 ratio) and halves 8.192 → 4.096 MHz, still a
  valid HSE. Field-test fallback, not a default part.
- **Sensor clocks** are generated by STM32 timers off the PLL (phase-locked to
  HSE, hence to the master): one **1.024 MHz** output to the ADXL355 (EXT_CLK),
  one **40.96 kHz** output to the ICM-42688 (CLKIN). No
  external divider ICs — at MHz rates they would draw about as much as the MCU
  saves.
- Because the MCU itself generates the sensor clocks, it **cannot enter Stop
  mode** (the timers would halt). It uses light **Sleep (WFI)** between DRDY
  interrupts. Acceptable: the always-on analog front end (3× LT3042 + sensors)
  dominates the power budget, so deep MCU sleep would save little anyway.

## D7 — Two protocols: a data plane and a control plane

The link layer splits into two independent protocols on the RS-485 data pair:

- **Data protocol (`nic-data`)** — the streaming sample packets during the
  token-ring data phase: `[1B station ID][2B sample index][present sensors ×
  3 axes × N bytes][2B CRC16]`. Minimal overhead, throughput-oriented, one
  packet per node per turn.
- **Control protocol (`nic-ctrl`)** — discovery / init handshake, address
  assignment and remapping, token-ring ordering, the broadcast sync packet, the
  calibration command, hard reset, and status / diagnostics. Addressed,
  acknowledged, low rate.

Keeping them separate keeps the data path lean while the control path can be as
chatty and robust as it needs without bloating every sample packet. (These two
are message **TYPEs** over one universal transport, not two libraries — see D13.)

## D9 — Control protocol (`nic-ctrl`): minimal, symmetric, opcode-driven, silence-framed

The control plane is deliberately tiny. It leans on the RS-485 line being idle
most of the time, and the **same frame works both directions** (master→node and
node→master). Exact byte order is not significant — the field set is what
matters:

- **Framing:** the bus sits silent between exchanges, so a frame start is found
  by UART **idle-line detection plus a 1-byte magic/sync marker** — no length
  prefixes, no complex headers.
- **Frame fields:** `SYNC, DST, SRC, TYPE, ARG` (+ optional reply byte, + 2B CRC16)
  - **DST** — destination. `0xFF` = broadcast. The network is <= 8 nodes by
    design, so one byte covers every node, the master, and broadcast.
  - **SRC** — sender's unit number, so the master knows which node a reply,
    error or status came from. A node→master frame just sets DST = master,
    SRC = its own number.
  - **TYPE / opcode** — what the packet means; drives interpretation of ARG.
    One byte = up to 255 opcodes, far more than needed.
  - **ARG** — opcode-specific (new address for ASSIGN_ADDR, error code for an
    error report, etc.).
  - **optional reply byte** — reserved for return queries / feedback.
- **Broadcast (DST = 0xFF)** carries global commands in one shot: "all reset",
  "all switch to HSE clock", "all calibrate", "all start sampling".
- **Opcodes** cover the whole lifecycle: DISCOVER/PING, ASSIGN_ADDR,
  SET_TOKEN_ORDER, SYNC, CALIBRATE, TOKEN_PASS, STATUS, ERROR, HARD_RESET.
- **CRC16** — 2-byte frame check on every control frame (see D10).

Both directions and both planes interlock: the master orchestrates with these
frames, and a node sends its `nic-data` packet only when granted the token
(TOKEN_PASS).

## D10 — CRC16 on both protocols

Both the data and the control protocol close each frame with a **2-byte CRC16**.
The STM32H503 has a **hardware CRC unit**, so the check costs almost nothing to
compute, and at 125 Hz with small packets two extra bytes are negligible. This
supersedes the earlier "CRC8 on the control plane only" idea: end-to-end
integrity on **both** planes, catching in-flight corruption that the UART echo
check (transmit side only) cannot see.

## D12 — One firmware for the node's lifetime; no over-bus firmware update

The node is intentionally dumb, so its firmware surface is tiny and all evolving
logic (processing, formats, analysis) lives on the master. Decision: ship **one
firmware version for the node's whole service life**, no firmware update over
the bus — the transport needs no bootloader/OTA opcodes.

Trade-off accepted with eyes open: a node is potted and buried, so it can never
be physically reflashed — the firmware must be correct before potting, because
there is no patch afterwards. This is the reason to keep the node firmware
minimal and heavily host-tested, which the modular mock-tested design targets.

- *Optional free safety net:* the STM32H503 has a built-in UART bootloader.
  Since the node already talks over a UART (RS-485), wiring BOOT0 so it can be
  entered gives a factory reflash path over the bus with **no firmware to
  write**. Pure hardware option.

On sensor state across power cycles: the ICM-42688 and ADXL355 have **no user
non-volatile config**, so their registers come up in the default/standby state
on every power-on. The node firmware therefore **reconfigures the sensors at
each boot** (the driver `init`: ODR, CLKIN/EXT_CLK, ranges, enable) — by design,
not extra work. The **calibration matrices persist in the STM32 flash**, so
levelling survives power cycles; only the volatile sensor config is re-applied.

## D13 — The comm library is a universal frame bridge; semantics live in the application

The transport library is **not** wired to any specific commands. It is a dumb,
symmetric, universal bridge: it frames, addresses and CRC-checks bytes in both
directions and hands the application a validated frame. It holds **no opcode
table and no protocol state machine** — it does not know what any TYPE means.

- One library (`nic-link`), **identical on master and node** (symmetric frame).
- The application layer (the master's software; the node's `nic-node` glue)
  decides what every TYPE/payload means. A new command needs no transport
  change — just a new TYPE and payload.
- Software embodiment of "the node is dumb": the node app only routes payloads
  to/from its sensors; all evolving protocol meaning is above the bridge.

The "two protocols" (D7) collapse into **two message TYPEs over this one
bridge**, not two libraries.

**Universal frame:** `[SYNC][DST][SRC][TYPE][LEN][payload×LEN][CRC16]`
- DST/SRC addressing (`0xFF` = broadcast), TYPE = application meaning, LEN =
  payload length (any short payload fits — e.g. sensor + register + value), 2-byte
  CRC16 over the frame.
- API is just `encode(dst, src, type, payload[]) -> bytes` and
  `decode(bytes) -> {dst, src, type, payload, crc_ok}`. Bytes in, bytes out —
  fully host-testable.

## D19 — Final framing: two lean frames (supersedes the universal frame of D13)

The wire format is **two purpose-built frames**, each started by its own magic
byte and delimited by the bus idle line (the magic also tells them apart):

- **DATA** (slave → master, 125×/s): `[MAGIC_D][station][index 2B][payload][CRC16]`.
  **No length field** — the payload length is fixed per node, agreed at the init
  handshake, and does not change in operation (absent sensors pad zeros). Lean,
  because the slot order already established whose turn it is.
- **CONTROL** (master ↔ slave, on demand): `[MAGIC_C][station][attribute][value][CRC16]`
  = 6 bytes. `attribute` = opcode, `value` = argument; the same frame serves both
  directions (a reply sets its own station).

This refines D7/D9/D13: still one library (`nic-link`), still "dumb transport,
meaning in the app", but now **two distinct lean frames** instead of one shared
envelope — the high-rate data path carries no addressing/length overhead it does
not need.

- **CRC-16-CCITT on both** (restores D10). One polynomial → the STM32 hardware
  CRC unit is configured once and shared; the control frame costs just 1 byte
  more than a CRC-8 would, not worth a second config + the shared-unit hazard.
- **Temperature rides in the DATA payload** (one byte per present sensor — each
  sensor die drifts on its own) — no separate housekeeping stream. Slow fields
  (tilt, temperatures) repeat their latest value; the redundancy is squeezed at
  storage (NIC-DMD / Steim), so the node never bothers suppressing repeats. See
  DATAFLOW.md for the byte layout.

## D20 — Runtime setting changes roll over to a new storage table

A setting (sensor range, address, …) can be changed in operation by a CONTROL
frame (attribute + value). When that happens, the **master closes the current
NIC-MLA table and opens a new one** stamped with the new settings; reconciliation
is left to the PC tooling. The node just applies the command — it keeps no
history. This keeps the node dumb and the on-disk record self-describing per
settings epoch.

## D23 — Node identity is the number on the box, not the MCU UID

Each node has a **permanent hardware number** (identity) and a **software address**
(its slot for the current deployment). The hardware number is a **small human-set
number printed on the enclosure** — 1, 2, 3 … — read at boot via `board_hw_addr()`.
We deliberately did **not** use the STM32's 96-bit factory UID as the identity.

Why the hand-set number wins:
- **A human remembers it.** "The box in the 100 m borehole is 6, the one 2 m down
  is 1." Nobody tracks a 96-bit hex string for a unit buried in rock.
- **Trivial to set.** It is one **build constant** — change it and reflash (the
  image is a few KB, done a handful of times in a unit's life). A flash cell +
  provisioning path would buy "one binary for all boxes", but that's
  over-engineering at this scale; `board_hw_addr()` keeps the seam so it can become
  flash-backed later if ever mass-produced.
- **Structure in the table.** On power-up or hard reset the node falls back to its
  hardware number and reports *that* — never a scrambled software address. So after
  any reset you always know *which physical box*, hence *which location*, a stream
  came from; you never log data from an anonymous source.

The master software-remaps each hardware number to a slot (`ASSIGN_ADDR`) from a
hardware → slot table it keeps; the software address **is** the token slot, so no
separate "set token order" opcode is needed. Permanent identity vs per-deployment
role is also what makes a box portable between installations. (Provisioning must
keep the numbers unique — a duplicate hardware number collides on the bus at boot.)

## D24 — Self-running TDMA slots + per-sample status byte

**Data runs itself.** The master distributes the clock, assigns each node a slot
(= its software address) at init, says go, then **only listens and asks on demand**
— it does *not* hand out a per-sample token. Each phase-locked node transmits in its
**own clock-derived slot**: it fires when it overhears the predecessor's frame finish,
or at its slot deadline (round start + (slot−1)×slot-width) if that node is silent.
Slots are disjoint → collision-free without arbitration; a missing node costs one
slot of silence and the ring carries on. This supersedes the master-polled token ring
of D7 (no TOKEN per sample; `TOKEN` survives only as an explicit repair / out-of-band
grant), and deletes the per-sample token overhead — the bus-load figure is DATA-only
(~272 kbps for 8 full nodes; step the RS-485 rate up for a full ring with turnaround
margin). Discovery and slot order key on the box hardware number (D23), so no
separate "set token order" opcode is needed.

**Per-sample status byte** — the fixed last payload byte (→ 28 B full payload). Eight
flags the node computes cheaply from values it already holds, so the master never does
diagnostics and an offline pass spots trouble at a glance:

| bit | flag |
|---|---|
| 0–3 | saturation of ADXL355 / ICM-accel / ICM-gyro / SCL3300 |
| 4 | input (bus) supply low |
| 5 | internal 5 V (switcher) low (< ~4.8 V — switcher failing / oscillating) |
| 6 | over-temperature — any die or the MCU > 50 °C |
| 7 | a present sensor went silent |

The saturation bits also drive the ADXL↔ICM stitching (D16): the master learns
*which* sensor railed without re-deriving it from the numbers. Dropped from earlier
drafts: an on-RC/desync bit (a DATA frame only exists when synced) and a link-error
bit (the master is the receiver — it sees link errors itself).

**Detail on demand** — kept off the per-sample byte, pulled only when the master
*asks*: `GET_VOLTAGE` (exact value), `SENSOR_TEST` (which sensors respond),
`GET_CPUTEMP` (exact MCU temperature). A critical failure needs no bit: a dead node
simply stops transmitting, the master sees the empty slot, fills zeros and marks it
KO in its table.

## D25 — Master uplink: LoRa out-only (+ resend), Wi-Fi optional & rich

Two uplink tiers, split by what each radio is good at:

- **LoRa — outbound telemetry only.** A fixed **16-byte** frame of *values*, not
  text warnings (a warning is just a value crossing a threshold; the receiver
  decides): master id, RTC time, supply voltage, temperature, seismic event count,
  a node-alive bitmask, the OR of the network's D24 status flags, and the last
  event peak — closed with CRC-16-CCITT (`nic-lora`). The **only** downlink is a
  **resend** request on bad reception (mirrors the RS-485 `RESEND_DATA`, one tier
  up): reliability, not control, so there is no command/config surface on a remote
  unit. Bidirectional LoRa is deliberately skipped — the EU 868 duty cycle barely
  allows it, and a downlink able to reconfigure a buried node is an attack surface
  for almost no benefit.
- **Wi-Fi — optional and rich, but fully decoupled.** Where Wi-Fi exists the master
  is mains-powered with bandwidth, so it carries what LoRa cannot: bulk NIC-MLA
  upload, NTP time, a small config/status API, and **master-only** OTA (the nodes
  stay one-firmware-for-life, D12). The network records and runs **without** Wi-Fi
  (SD store-and-forward) — a convenience, never a dependency.

Net: LoRa is the always-on heartbeat/alert beacon; Wi-Fi is "do everything when
connected". Each does only what its physics is good for.

**Transport- and payload-agnostic.** "Wi-Fi" is the *rich-uplink role*, not the wire:
Ethernet (an ESP32 RMII PHY, or a W5500) or cellular fill it identically — nothing
below the master changes. The payload is a free choice too: ship the **native MLA
container, DMD-compressed as it sits**, straight to the server (lean, nothing to
re-encode), or stream **SeedLink** (real-time miniSEED over TCP — the `mseed` export
already makes the miniSEED) for FDSN/IRIS consumers. The network speaks the standard
when something wants it, without being forced to convert; the native format travels
onward just as well.

## D26 — Node TYPE: a separate identity field, learned at discovery

A node has **two** identities: its **NUMBER** (which box — D23) and its **TYPE** (what
kind — seismo / Basic / iono / starDust). They are orthogonal: the number is unique
per box and human-set ("the box in the borehole is 6"); the type is a per-**product**
build constant shared by every box of a line. The number says *which physical unit /
location*; the type says *what schema and role*, so the master applies the right decode
and uplink to each stream.

Why type is its **own** field, not inferred:
- **Payload length doesn't carry it.** A fully-populated seismo node and a Basic
  weather node are **both 28 B** — length cannot tell them apart. `STATUS` reports
  length (needed to decode the frame, D19); type is a distinct question.
- **Reported at discovery, stored against the number.** The node answers an
  `NIC_OP_NODE_TYPE` query with its `CFG_NODE_TYPE`; the master records it next to the
  box number. On a master restart, discovery re-learns every (number → type) mapping
  fresh — nothing relies on persistence.
- **Kept off the DATA frame.** The high-rate sample path stays lean (D19/D24): the
  station number in each frame already keys back to the type via the master's table, so
  a per-sample type byte would be 125×/s of a constant. Type lives in the control plane,
  learned once.

The **seismo** node is exactly the NIC-Quake node — same hardware, same firmware. Only
two constants ever differ per unit: the box **NUMBER** (per box) and, for a non-Quake
line that reuses this firmware, **`CFG_NODE_TYPE`** (per product) plus its sensor front.
Everything else — and thus full bus compatibility — stays identical.

Type codes: `NIC_NODE_TYPE_SEISMO` (1), `…_BASIC` (2), `…_IONO` (3), `…_STARDUST` (4);
0 = unknown / not yet reported (`nic_proto.h`).

## D27 — Absolute time: a GPS-disciplined anchor + the sample index (no per-sample timestamps)

Two layers of time, from two sources:

- **Relative (nodes on one bus)** comes from the **shared 8.192 MHz clock**. Every node is
  phase-locked to it, so their samples are co-timed to **sub-microsecond**; the 2-byte
  sample **index** pins each sample's sub-second position identically on every node. Free,
  and the whole point of the common clock (D5/D6).
- **Absolute / global** comes from **GPS**. The satellites' atomic clocks are one worldwide
  timescale; a GNSS receiver gives the absolute second and a **PPS** pulse marks its exact
  edge (~ns).

**Storage model: anchor + index, never a timestamp per sample.** Because the 125 Hz clock
is disciplined, the samples are *exactly* evenly spaced — sample K of second N is at
N + K/125 s, guaranteed by hardware, not assumed. So a record stores the time **once per
second** (the anchor) plus the rate, and every sample's time is **computed**. This is
miniSEED practice (record = start time + sample rate; individual samples carry no time) and
it keeps the stream lean.

**The anchor ties the free-running clock to GPS, once per second.** At each PPS edge the
master records *where the network clock / sample index was* — binding the otherwise
free-running 8.192 MHz TCXO to absolute GPS time. So `timestamp` (the second) ← GPS,
`subsec` (position in the second) ← the sample index, and the **(GPS-second ↔ index) tie
recorded every second corrects the slow TCXO drift every second**. This is the "RTC second +
GPS PPS + NIC-clock interpolation" of the Iono node.

**Why this gives _global_ sync.** Each station disciplines its own clock to its own GPS, and
all GPS share one time — so "GPS second 1234567890, sample 42/125" is the **same absolute
instant in Tokyo and in Prague**. Stations are not synced to each other by passing time
around; each is independently locked to the same satellites. Two layers: the master's
**SYNC aligns nodes within a bus**, **GPS aligns buses globally**. That is what lets one
sample be lined up across a worldwide grid of stations.

**The GPS node is the exception — and the reference.** Its sensor (the receiver) inherently
knows absolute time, so GNSS observables are **self-timestamped in the epoch header** — and
that self-timestamp is exactly what disciplines the network anchor. The dumb nodes (seismic,
meteo) get their time **computed** from the anchor; the GPS node *carries* it. Both
encodings coexist, each where it fits.

**Consequence — global absolute time needs a GPS/PPS at the master.** Relative timing is
free everywhere (the shared clock); absolute global timing requires a PPS source. A
Quake-only station with just an RTC has excellent relative timing but its absolute time
drifts (RTC crystal) — to join the global grid it adds a PPS source: the Iono UM980, or even
a cheap PPS-only GPS used purely for time. Nodes stay dumb throughout: they never know
wall-clock time, only "samples since the last TICK" — the **master owns absolute time**
(D5/D12).

Quality budget: PPS ~tens of ns, inter-node sub-µs, TCXO interpolation over one second ~µs →
**sub-millisecond, realistically single-digit µs** absolute timing worldwide — ample for
seismology (P-wave onsets, cross-correlation) and for ionospheric TEC. Storage is already
shaped for it: NIC-MLA carries `timestamp` + `subsec`; samples carry no time.

**Mechanism (recording level), and why the crystal barely matters.** The sampling
oscillator free-runs, so the number of ticks (hence samples) in a real GPS second wobbles
by its ppm — at 1 ppm a whole sample is gained or lost only every ~8000 s (~2 h), so most
seconds hold exactly 125 samples and the odd one holds 124/126. You do **not** force every
second to 125; you **measure** the wobble:

- At each **PPS edge** the master latches the **fast-clock counter** and pairs it with the
  PPS's absolute time → an anchor `(ticks ↔ ns)`. The absolute time of any sample is a
  **linear interpolation between the two latest anchors**; the slope *is* the measured true
  rate, so the ppm is absorbed, not approximated. (`nic-timesync` is this core, host-tested;
  the HW capture + NMEA/UBX read is glue.)
- **Accuracy comes from the fast-clock capture (~one 8.192 MHz tick, ~120 ns), not the
  sample rate** — latch PPS against the fast clock, never against the slow 125 Hz grid. A
  higher sample rate buys *resolution*, not timing accuracy.
- **The crystal's ppm hardly affects absolute accuracy**, because PPS re-anchors every
  second: even a sloppy 50 ppm part is bounded to ~50 µs *within* a second. A TCXO is still
  worth it — but for **GPS holdover** (drift if the fix drops: 1 ppm ≈ 3.6 ms/h, 50 ppm ≈
  180 ms/h) and **low jitter** (clean sampling), *not* for the headline accuracy, which PPS
  owns. The optional oscillator→MCU tap (D6) is what continuously **measures** the crystal
  against GPS (`nic_timesync_drift_ppb`), so a measured mediocre part beats an unmeasured
  good one.

**Per stream:** the **GPS node** is self-timestamped (its epoch header already carries GPS
time) → no interpolation. **Seismic** gets the interpolation above. **Meteo** is slow
enough that it only needs the second-level label so the archive stays searchable — even
seconds of error are harmless. The five GPS addresses reassemble into **one** logical
table; the dumb sensors each get their own.

**Optional upgrade — GPSDO.** Replace the fixed TCXO with a steerable VCTCXO and lock it to
PPS in a loop; then every second holds *exactly* N samples and no interpolation is needed.
Because one oscillator feeds the whole network, this frequency-locks every node to GPS at
once. More hardware (VCTCXO + DAC + loop) — an upgrade, not the default; "measure, don't
steer" is cheaper and already sub-ms.

**Anchor the time, not the clock — and why the choice barely matters.** Two ways tie
samples to GPS: *anchor the time* — let the oscillator free-run and write each
record's true GPS time as it lands (the default, `nic-timesync`) — or *anchor the
clock* — steer it so every second holds exactly N samples (the GPSDO above). But the
record already carries `timestamp` + `subsec`, so **the data is correctly placed
either way**; the only difference is whether the per-second sample *count* is constant
(steered) or wobbles by ±1 (free-running). A sample on a second boundary is a ±1
labelling question, not a timing error — its instant is the same on both sides and
miniSEED's per-record start time absorbs it. Correctness needs no disciplined clock;
uniform counts are a convenience, so the working default is "just write the time" and
let it float. And because every node shares the one central clock, absolute drift
shifts the whole record *uniformly* — it never distorts the inter-station structure
that onset-picking and cross-correlation read; sharper absolute time only sharpens
that downstream matching, it is not a correctness need. (If ever steered, the actuator
need not be analog: an on-chip fractional-N PLL — FRACN — or a digitally-trimmed
oscillator does it with no DAC.)

**DMD keyframes are dense anchors.** When the stream is DMD-compressed a keyframe falls
every few samples (≈ one in eight in the seismic profile). Each keyframe is a natural
re-anchor, so true time is re-pinned far more often than once per second; inter-anchor
drift is bounded by the keyframe interval (tens of ms of samples) and the timing can
simply float — write the time as the data arrives, no alignment.

**Sync-source budget.** *Relative* timing within a bus is the shared clock — sub-µs
whatever the absolute source. *Absolute* timing depends on that source: a **GPS/PPS**
anchor gives sub-ms (single-digit µs); **WiFi NTP/SNTP** — free on the ESP32 head-end —
gives ms on a LAN, tens of ms over the internet (sub-ms only with PTP and MAC-level
hardware timestamps); **BT/BLE** lands ms-to-tens-of-ms, useful mainly as a local or
setup-time sync; and a **LoRa-only** occasional sync gives the least.
Its error is the crystal drift between syncs plus LoRa's delivery jitter — a measured
2 ppm part, drift-corrected, holds ms-level over an hour; uncorrected ~7 ms/h, a plain
20 ppm crystal ~70 ms/h; and a LoRa timestamp not hardware-stamped on a defined symbol
adds tens of ms. Net: tens-to-hundreds of ms absolute — ample for meteo and single-bus
relative work, marginal for multi-station seismic location, which is why absolute
precision wants a PPS source.

**One GPS, route its PPS separately.** A station needs a single GPS — more receivers
buy nothing (independent units agree only to tens of ns, and the shared clock already
gives sub-µs *relative* timing). That one PPS is routed to whatever consumes it: a GPS
node runs on the distributed clock, so it can capture PPS against that clock locally and
ship the `(tick ↔ UTC)` anchor as ordinary node data — the raw edge never crosses the
bus; or, where the edge must reach a head-end steerer, it goes on its own conductor (a
dedicated line or a spare RS-485 pair), a fixed cable delay (~5 ns/m, tens of ns over
temperature) that calibrates out once. Module choice and the (basic, lightly
protected) RS-485 wrap are in `blocks/gps-pps.md`.

**Recommendation.** Run the software default — shared clock for relative sync, write
each record's true time, measure-and-correct the drift (sub-ms with a PPS source).
Everything above about *steering* the clock — GPSDO, FRACN, the dedicated-PPS plumbing —
is a documented option, not a requirement: build it only where a deployment wants
exactly-uniform seconds.

## D28 — Generic config slots (front-agnostic protocol, front-defined meaning)

A node's runtime-tunable settings (D21) ride a reserved block of **generic config-slot
opcodes** instead of one named opcode per sensor: the protocol knows only "set slot N to
value V" (`NIC_OP_CFG(slot)`; the value byte is the setting). **What each slot means is
defined by the node's FRONT glue**, not by the shared protocol.

Why, in the multi-front platform: if every front named its own opcodes, the shared
`nic_proto.h` would accumulate seismo's four + iono's + starDust's … — exactly the per-front
leak the universal core must avoid. Generic slots keep the shared protocol **front-agnostic**
(a new front adds zero protocol opcodes); each front's glue maps its own slots — e.g. seismo:
slot 0 = ADXL range, 1 = ICM accel, 2 = ICM gyro, 3 = SCL mode. Same "dumb transport, meaning
in the application" as D13/D19, on the control plane.

- **No frame change.** The slot rides in the **attribute** byte (reserved range 32..39 =
  slots 0..7), because the lean control frame's single value byte can't hold both slot and
  value. The ACK echoes the slot opcode, so the master knows which slot landed; the
  settings-epoch roll (D20) stamps "slot X = value Y" with no need for semantics (PC tooling
  reconciles meaning per epoch).
- **Readability lives in the glue, not the wire:** each front defines named constants
  (`SEISMO_CFG_ICM_GYRO 2`), so code still reads `set_param(…ICM_GYRO, 15)`.
- **Discipline (D21 still binds):** slots carry sensor sensitivity/range only — not a
  backdoor to a smart node.

Supersedes the four named `SET_*` opcodes (concept stage, nothing deployed).
